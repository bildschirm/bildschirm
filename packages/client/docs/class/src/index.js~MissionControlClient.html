    MissionControlClient

    The mission control client class.

    You can easily build your own client implementation, this one is just easy to use and has everything you might need.

    The client hides the authentication implementation details from the user, by hijacking the connect event. The connect event only gets called after the authentication scheme is successful.

    Execute an action on the mission control server.

    This method sends an 'action' event, which the server will use to execute the action and modify the state accordingly.

    Params:

    action string

    The action that you want to execute.
    data object

    Example:

    client.action('EXAMPLE:DO', { parameter: 'example' })
client.action('VIDEO-QUEUE:PUSH', { video: { url: '...', format: 'mp4' }})

    public subscribe(serverEvent: string, listener: function(data: object)): function since 1.0.0 source

    Subscribe to a server event (actions, state updates).

    To subscribe to a server event, we need to emit a 'subscribe' event to the server so it knows to broadcast the right events to us. This function automatically handles these 'subscribe' and 'unsubscribe' events so you can simply use this method to do it. When we disconnect from the server, this function also handles resubscribing to the events. It returns a function that can you can use to remove the event listener again and unsubscribe from the server.

    Params:

    NameTypeAttributeDescription
    serverEvent string

    This is the server event you want to subscribe to. Keep in mind these are not general socket events, but rather state / action Mission Control events.
    listener function(data: object)

    The listener function that will be called on event.

    Return:

    function

    Returns a function which you can use to remove the event listener.

    Example:

    client.subscribe('action:EXAMPLE:DO', (data) => {})
client.subscribe('update:stateObject', (data) => {});

    private _setupSocketHandlers() source

    This function sets up all the listeners for the socket (connect, disconnect, error, reconnect, etc).

    Job of this function is to unify all error events into a shape that makes more sense. See SOCKET_ERROR for the possible errors.
